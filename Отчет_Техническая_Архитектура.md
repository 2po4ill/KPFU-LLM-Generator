# Основная часть

## 1. Общая архитектура системы

### 1.1 Модульная структура системы

Целью разработки является создание системы автоматической генерации образовательного контента на основе больших языковых моделей для Казанского Федерального Университета. На момент начала работы существовали отдельные решения для обработки PDF-документов и взаимодействия с LLM, однако они не были интегрированы в единую систему и не обеспечивали требуемого качества генерируемого контента. Система должна обрабатывать учебную литературу из библиотеки КФУ, извлекать релевантные фрагменты по заданной теме и генерировать структурированные лекционные материалы, соответствующие стандартам ФГОС. Для обеспечения масштабируемости и поддерживаемости решения необходимо было спроектировать модульную архитектуру с четким разделением ответственности между компонентами, где каждый модуль отвечает за свою область функциональности и может развиваться независимо.

Система построена по принципу модульной архитектуры с четким разделением ответственности между компонентами.

**Диаграмма модулей:**

**Принципы архитектуры:**

1. **Separation of Concerns** - каждый модуль отвечает за свою область
2. **Dependency Injection** - зависимости передаются через конструкторы
3. **Single Responsibility** - один модуль = одна ответственность
4. **Loose Coupling** - модули слабо связаны через интерфейсы
5. **High Cohesion** - внутри модуля высокая связность

## 2. Архитектура выбора релевантного контента

### 2.1 Сравнительный анализ подходов

При проектировании механизма выбора релевантных фрагментов из учебной литературы были рассмотрены два фундаментально различных подхода.

**Таблица 5. Сравнение архитектурных подходов**

| Критерий | Chunk-based (Семантический поиск) | TOC-based (Анализ оглавления) |
|----------|-----------------------------------|-------------------------------|
| **Архитектурный принцип** | Bottom-up: от фрагментов к целому | Top-down: от структуры к деталям |
| **Технологии** | FAISS, sentence-transformers, векторные операции | LLM, простой текстовый анализ |
| **Предобработка** | Разбиение на chunks (1000 символов), векторизация, индексация | Извлечение оглавления (страницы 3-7) |
| **Поиск** | Cosine similarity в векторном пространстве | Семантический анализ LLM |
| **Результат** | Топ-K наиболее похожих chunks | Номера страниц целых разделов |
| **Точность** | Средняя (находит упоминания, не обязательно детальный материал) | Высокая (находит целые разделы по теме) |
| **Скорость** | ~60 секунд (векторизация + поиск по индексу) | ~7 секунд (один запрос к LLM) |
| **Память** | Высокая (10,000+ векторов × 384 dim = ~15MB + индекс) | Низкая (только текст оглавления ~5KB) |
| **Сложность** | Высокая (FAISS, векторные операции, управление индексом) | Низкая (один промпт к LLM) |
| **Контекст** | Фрагментированный (границы chunks произвольные) | Полный (естественные границы разделов) |
| **Масштабируемость** | O(log n) для поиска, O(n) для индексации | O(1) для поиска (размер TOC фиксирован) |

**Экспериментальные результаты (тема "Основы ООП"):**

| Подход | Выбранные страницы | Содержание | Релевантность | Полнота |
|--------|-------------------|------------|---------------|---------|
| Chunk-based | 58, 99, 112 | Краткие упоминания ООП в разных контекстах | 40% | 30% |
| TOC-based | 101-112 | Полная глава "Объектно-ориентированное программирование" | 100% | 95% |

**Архитектурная диаграмма Chunk-based подхода:**

```
PDF Книга
    ↓
Извлечение текста (PyMuPDF)
    ↓
Разбиение на chunks (1000 символов)
    ↓
Векторизация (SentenceTransformer)
    ↓
Индексация (FAISS)
    ↓
Запрос (тема лекции) → Векторизация
    ↓
Поиск Top-K похожих chunks (cosine similarity)
    ↓
Результат: Фрагменты текста
```

**Архитектурная диаграмма TOC-based подхода:**

```
PDF Книга
    ↓
Извлечение страниц 3-7 (оглавление)
    ↓
Промпт к LLM: "Какие страницы относятся к теме X?"
    ↓
LLM анализирует структуру и возвращает номера страниц
    ↓
Загрузка полных страниц по номерам
    ↓
Результат: Полные страницы с контекстом
```


### 2.2 TOC-based решение и его преимущества

На основе сравнительного анализа было принято решение использовать TOC-based архитектуру.

**Технические преимущества:**

1. **Семантическое понимание структуры**
   - LLM понимает иерархию разделов
   - Учитывает контекст всего оглавления
   - Различает основные разделы и подразделы

2. **Производительность**
   - 8.5x ускорение (7с vs 60с)
   - Линейная сложность O(1) вместо O(log n)
   - Не требует предварительной индексации

3. **Качество контекста**
   - Полные страницы вместо фрагментов
   - Естественные границы (начало/конец раздела)
   - Сохранение авторской структуры изложения

4. **Простота архитектуры**
   - Отказ от FAISS (упрощение зависимостей)
   - Отказ от chunking (упрощение препроцессинга)
   - Один промпт вместо сложного пайплайна

**Таблица 6. Архитектура промпт-инжиниринга**

| Аспект | Сложный промпт | Простой промпт |
|--------|----------------|----------------|
| **Длина** | 15+ строк инструкций | 3-5 строк |
| **Структура** | Множественные правила, условия, исключения | Естественный вопрос |
| **Инструкции** | Негативные ("НЕ делай X") | Позитивные примеры |
| **Примеры** | Абстрактные | Из реального контекста |
| **Результат** | Путаница (номера разделов вместо страниц) | Корректные номера страниц |
| **Успешность** | 0-20% | 95-100% |

**Финальная архитектура промпта:**

```
Принципы:
1. Естественный язык (как человек спросил бы)
2. Позитивные примеры (что нужно получить)
3. Одна задача на промпт (Single Responsibility)
4. Примеры из реальных данных (не абстрактные)

Структура:
┌─────────────────────────────────────┐
│ Контекст: Текст оглавления          │
├─────────────────────────────────────┤
│ Задача: Простой вопрос              │
├─────────────────────────────────────┤
│ Формат: Позитивные примеры          │
├─────────────────────────────────────┤
│ Выход: Номера страниц или "0"       │
└─────────────────────────────────────┘
```

**Технические детали реализации:**

```python
# Псевдокод архитектуры TOC-based выбора

class LiteratureProcessor:
    def select_pages_from_toc(self, theme: str, book_id: int):
        # 1. Загрузка PDF
        pdf = self.load_pdf(book_id)
        
        # 2. Извлечение оглавления (страницы 3-7)
        toc_pages = pdf.pages[2:7]  # 0-indexed
        toc_text = self.extract_text(toc_pages)
        
        # 3. Определение offset (см. раздел 3)
        offset = self.detect_page_offset(pdf)
        
        # 4. Запрос к LLM
        prompt = self.build_toc_prompt(toc_text, theme)
        response = self.llm.generate(prompt)
        
        # 5. Парсинг ответа
        book_page_numbers = self.parse_page_numbers(response)
        
        # 6. Применение offset
        pdf_page_numbers = [p + offset for p in book_page_numbers]
        
        # 7. Загрузка полных страниц
        pages = [pdf.pages[p] for p in pdf_page_numbers]
        
        return pages
```

**Обработка граничных случаев:**

| Случай | Обработка |
|--------|-----------|
| Тема не найдена в оглавлении | LLM возвращает "0", книга пропускается |
| Оглавление на нестандартных страницах | Поиск по ключевым словам ("Содержание", "Оглавление") |
| Несколько разделов по теме | LLM возвращает все релевантные диапазоны |
| Очень большой раздел (>50 страниц) | Ограничение на 50 страниц, приоритет первым страницам |
