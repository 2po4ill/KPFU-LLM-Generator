# Основная часть

## 1. Архитектура системы выбора релевантных страниц

### 1.1 Сравнительный анализ подходов к поиску контента

При проектировании системы автоматической генерации образовательного контента ключевой задачей является выбор релевантных фрагментов из учебной литературы. Были рассмотрены два фундаментально различных архитектурных подхода.


**Таблица 1. Сравнение архитектурных подходов к выбору контента**

| Критерий | Семантический поиск (Chunk-based) | Анализ оглавления (TOC-based) |
|----------|-----------------------------------|-------------------------------|
| **Принцип работы** | Разбиение текста на фрагменты (chunks) по 1000 символов, векторизация через embeddings, поиск через FAISS | Извлечение оглавления книги, анализ LLM для определения релевантных разделов и номеров страниц |
| **Преимущества** | • Детальный поиск по всему тексту<br>• Не зависит от структуры книги<br>• Находит упоминания темы в любом месте | • Понимает структуру книги<br>• Работает с любым форматом оглавления<br>• Возвращает полные страницы с контекстом<br>• Быстрая обработка |
| **Недостатки** | • Фрагментация контекста<br>• Высокая вычислительная сложность<br>• Может выбирать краткие упоминания вместо детальных разделов<br>• Требует индексации (FAISS) | • Зависит от качества оглавления<br>• Не найдет материал, не указанный в оглавлении<br>• Требует LLM для анализа |
| **Точность выбора** | Средняя (выбирает страницы с упоминаниями, не обязательно с детальным материалом) | Высокая (выбирает целые разделы, посвященные теме) |
| **Время обработки** | ~60 секунд (векторизация + поиск) | ~7 секунд (прямой запрос к LLM) |
| **Сложность реализации** | Высокая (FAISS, векторные операции, индексация) | Низкая (один промпт к LLM) |
| **Использование памяти** | Высокое (хранение 10,000+ векторов) | Низкое (только текст оглавления) |
| **Качество контекста** | Фрагментированный (chunks по 1000 символов) | Полный (целые страницы) |

**Результаты тестирования на примере темы "Основы ООП":**

| Подход | Выбранные страницы | Содержание | Оценка |
|--------|-------------------|------------|--------|
| Chunk-based | 58, 99, 112 | Краткие упоминания ООП в разных контекстах | ❌ Неоптимально |
| TOC-based | 101-112 | Полная глава "Объектно-ориентированное программирование" | ✅ Корректно |


### 1.2 Обоснование выбора TOC-based решения

На основе сравнительного анализа было принято решение использовать архитектуру на основе анализа оглавления (TOC-based approach) по следующим причинам:

**1. Семантическое понимание структуры**
- Оглавление представляет авторскую структуру материала
- LLM понимает связь между темой лекции и разделами книги
- Выбираются целые разделы, а не фрагменты

**2. Производительность**
- 8.5x ускорение по сравнению с векторным поиском (7с vs 60с)
- Не требует предварительной индексации
- Линейная сложность O(n) вместо O(n log n)

**3. Качество контекста**
- Полные страницы сохраняют контекст изложения
- Отсутствие артефактов разбиения на chunks
- Естественные границы разделов

**4. Простота архитектуры**
- Отказ от векторной базы данных (FAISS)
- Отказ от процесса чанкинга
- Единый промпт для анализа

**Архитектурное решение:**

```
Вход: Тема лекции + Текст оглавления (страницы 3-7)
       ↓
Процесс: LLM анализирует оглавление и определяет релевантные разделы
       ↓
Выход: Список номеров страниц ИЛИ "0" (если тема не покрыта)
       ↓
Загрузка: Полные страницы с контентом
```

**Принципы промпт-инжиниринга:**

Было установлено, что эффективность промпта обратно пропорциональна его сложности:

| Тип промпта | Характеристики | Результат |
|-------------|----------------|-----------|
| Сложный | • 15+ строк инструкций<br>• Множественные правила<br>• Негативные инструкции ("НЕ делай X") | Низкая точность (путаница между номерами разделов и страниц) |
| Простой | • 3-5 строк<br>• Естественный вопрос<br>• Позитивные примеры | Высокая точность (корректные номера страниц) |

**Финальная архитектура промпта:**
- Естественный язык вместо формальных правил
- Позитивные примеры вместо негативных ограничений
- Одна задача на промпт (Single Responsibility Principle)
- Примеры из реального контекста данных

## 2. Архитектура обработки PDF и синхронизации нумерации

### 2.1 Проблема несоответствия номеров страниц

В процессе проектирования была выявлена фундаментальная проблема: номера страниц в оглавлении книги (колонтитулы) не соответствуют номерам страниц в PDF-файле.

**Математическая модель проблемы:**

```
PDF_page_number = Book_page_number + offset

где offset определяется структурой книги:
- Обложка
- Титульные страницы  
- Предисловие
- Оглавление
```

**Таблица 2. Влияние проблемы на точность системы**

| Метрика | Без учета offset | С учетом offset | Изменение |
|---------|------------------|-----------------|-----------|
| Точность выбора страниц | 0% (неправильные страницы) | 100% (корректные страницы) | +100% |
| Точность контента | 0% (галлюцинации) | 82.4% (верифицировано) | +82.4% |
| Использование книжного материала | Нет (LLM игнорирует нерелевантный контекст) | Да (LLM использует предоставленные страницы) | Критическое |
| Качество лекций | Неприемлемое | Производственное | Качественный скачок |

**Пример несоответствия:**
- Оглавление: "7.4 Строки ... страница 36"
- PDF страница 36: содержит "6.3 Выбор редактора"
- PDF страница 44: содержит "7.4 Строки" (36 + offset 8)


### 2.2 Универсальное решение для определения смещения

Разработана архитектура автоматического определения offset, работающая для любых книг без ручной конфигурации.

**Алгоритм определения смещения:**

```
1. Сканирование первых 50 страниц PDF
   ↓
2. Поиск колонтитула с номером "1" (первая пронумерованная страница)
   ↓
3. Вычисление: offset = PDF_page_number - 1
   ↓
4. Применение: PDF_page = Book_page + offset
```

**Таблица 3. Стратегии поиска первой страницы**

| Стратегия | Описание | Надежность | Применимость |
|-----------|----------|------------|--------------|
| Поиск в footer | Поиск изолированной цифры "1" в последних строках | Высокая | Стандартные книги с нижними колонтитулами |
| Поиск в header | Поиск в первых строках страницы | Средняя | Книги с верхними колонтитулами |
| Паттерн-матчинг | Поиск паттернов типа "Стр. 1", "Page 1" | Средняя | Книги с текстовыми метками |
| Изолированные цифры | Поиск одиночных цифр в начале/конце страницы | Низкая | Резервный метод |

**Архитектурные характеристики решения:**

**Универсальность:**
- Работает с различными форматами колонтитулов
- Адаптируется к разным структурам книг
- Не требует ручной настройки для каждой книги

**Надежность:**
- Каскад стратегий поиска (fallback mechanism)
- Валидация найденного offset
- Логирование для отладки

**Производительность:**
- Время выполнения: <1 секунда
- Кэширование результата для повторного использования
- Выполняется один раз при загрузке книги

**Интеграция:**
- Прозрачна для остальной системы
- Автоматически вызывается при обработке PDF
- Результат используется во всех операциях с номерами страниц

## 3. Архитектура генерации образовательного контента

### 3.1 Сравнение стратегий генерации

При проектировании процесса генерации лекционного материала были рассмотрены три архитектурных подхода, различающихся по количеству этапов и стратегии обработки контента.

**Таблица 4. Сравнение архитектур генерации контента**

| Критерий | Однопроходная (V1) | Двухэтапная (V2) | Многофазная с батчингом (V3) |
|----------|-------------------|------------------|------------------------------|
| **Принцип** | Один запрос к LLM со всеми страницами | Этап 1: Структура<br>Этап 2: Детализация | Фаза 1A: Извлечение концепций (батчи)<br>Фаза 1B: Детализация (батчи)<br>Фаза 2: Разделы (параллельно) |
| **Средняя длина** | 312 слов | 2,130 слов | 2,941 слов |
| **Время генерации** | ~450 секунд | ~280 секунд | ~198 секунд |
| **Обработка страниц** | 5-10 страниц | 30 страниц | 47 страниц |
| **Покрытие концепций** | ~40% | ~95% | ~100% (с дедупликацией) |
| **Структурированность** | Низкая | Высокая | Очень высокая |
| **Использование GPU** | Пиковое (100%) | Высокое (80-90%) | Оптимальное (50-60%) |
| **Проблема дубликатов** | Нет (мало контента) | Есть (повторы концепций) | Решена (дедупликация) |
| **Ограничение контекста** | Да (превышение 8K токенов) | Частично | Нет (батчинг) |

**Эволюция подходов:**

**V1 (Однопроходная):**
- Простая архитектура: один промпт → одна лекция
- Проблема: недостаточная глубина, короткий контент
- Причина: LLM пытается уместить весь материал в один ответ

**V2 (Двухэтапная):**
- Разделение на планирование и исполнение
- Улучшение: структурированность, длина контента
- Проблема: дублирование концепций, ограничение по страницам

**V3 (Многофазная с батчингом):**
- Модульная обработка с параллелизацией
- Решение всех предыдущих проблем
- Оптимальный баланс качества и производительности


### 3.2 Многофазная архитектура с батчингом (Generator V3)

Финальная архитектура представляет собой многофазный пайплайн с параллельной обработкой и интеллектуальной дедупликацией.

**Архитектурная схема:**

```
Вход: 47 страниц книги
    ↓
┌─────────────────────────────────────────┐
│ Фаза 1A: Извлечение концепций (Batched) │
│ • Разбиение на батчи по 5 страниц       │
│ • Параллельная обработка (2 батча)      │
│ • Извлечение 5-8 концепций/батч         │
│ Выход: 73 сырых концепции (~41с)        │
└─────────────────────────────────────────┘
    ↓
┌─────────────────────────────────────────┐
│ Фаза 1A.5: Дедупликация концепций       │
│ • Нормализация (lowercase, trim)        │
│ • Удаление точных дубликатов            │
│ • Слияние похожих (substring, 80%)      │
│ Выход: 32 уникальных концепции (<1с)    │
└─────────────────────────────────────────┘
    ↓
┌─────────────────────────────────────────┐
│ Фаза 1B: Детализация концепций (Batched)│
│ • Балансировка: 6+6+6+6+4+4 концепций   │
│ • Параллельная обработка (2 батча)      │
│ • Генерация 400-500 слов/батч           │
│ • Встраивание примеров в концепции      │
│ Выход: 2,463 слова (~112с)              │
└─────────────────────────────────────────┘
    ↓
┌─────────────────────────────────────────┐
│ Фаза 2: Генерация разделов (Parallel)   │
│ • Введение (параллельно)                │
│ • Заключение (параллельно)              │
│ Выход: ~478 слов (~45с)                 │
└─────────────────────────────────────────┘
    ↓
Итого: 2,941 слов за 198 секунд
```

**Таблица 5. Ключевые архитектурные решения V3**

| Решение | Обоснование | Результат |
|---------|-------------|-----------|
| **Батчинг по 5 страниц** | Оптимальный баланс между размером контекста и детальностью извлечения | Эффективное использование контекстного окна LLM |
| **Параллельная обработка (2 батча)** | Максимальное использование GPU без перегрузки VRAM | 2x ускорение при стабильной нагрузке |
| **Дедупликация концепций** | Устранение повторов из разных страниц | 56% сокращение (73→32), нулевая избыточность |
| **Балансировка батчей** | Равномерное распределение нагрузки (6+6+6+6+4+4 вместо 6+6+6+6+6+6+1) | Оптимальное использование ресурсов |
| **Встраивание примеров** | Теория + практика в одной концепции вместо отдельной секции | Лучшая педагогика, экономия времени |
| **Использование ### заголовков** | Структурированный markdown вместо нумерации | Профессиональное форматирование, отсутствие конфликтов |

**Алгоритм дедупликации:**

```
Стратегия 1: Точное совпадение (case-insensitive)
    ↓
Стратегия 2: Substring matching (одна концепция содержится в другой)
    ↓
Стратегия 3: Word overlap (>80% общих слов)
    ↓
Результат: Уникальные концепции без потери информации
```

**Педагогическое преимущество встраивания примеров:**

| Подход | Структура | Эффект |
|--------|-----------|--------|
| Раздельный | Теория → Практика (отдельная секция) | Переключение контекста, потеря связи |
| Встроенный | Концепция → Пример → Объяснение | Немедленное подкрепление, лучшее запоминание |

## 4. Архитектура системы валидации качества

### 4.1 Методы предотвращения галлюцинаций

Большие языковые модели склонны к генерации правдоподобной, но фактически неверной информации. Для образовательного контента это критическая проблема, требующая архитектурного решения.

**Таблица 6. Сравнение подходов к валидации**

| Подход | Принцип | Преимущества | Недостатки | Точность |
|--------|---------|--------------|------------|----------|
| **Без валидации** | Доверие к LLM | Простота, скорость | Высокий риск галлюцинаций | Неизвестна |
| **Валидация против чанков** | Проверка утверждений против FAISS-индекса | Использует существующую инфраструктуру | Проверяет не те данные, что использовались для генерации | 0% (ложные негативы) |
| **Валидация против использованных страниц** | Проверка утверждений против фактически использованных страниц | Проверяет правильный источник, высокая точность | Требует хранения исходных страниц | 82.4% (верифицировано) |
| **Ручная проверка** | Экспертная оценка | 100% точность | Не масштабируется, требует времени экспертов | 100% |

**Выбранная архитектура: Валидация против использованных страниц**

**Принцип работы:**

```
Сгенерированная лекция
    ↓
Извлечение фактических утверждений (claims) через LLM
    ↓
Векторизация утверждений и исходных страниц
    ↓
Вычисление cosine similarity для каждого утверждения
    ↓
Утверждение подтверждено если max_similarity > 0.4
    ↓
Confidence = Подтвержденные / Всего утверждений
```

**Ключевое архитектурное решение:**
Валидация проводится против тех же страниц, которые были переданы LLM для генерации, а не против всей базы данных или чанков.

**Обоснование:**
- Логическая согласованность: проверяем то, что использовали
- Высокая точность: нет ложных негативов
- Эффективность: малый объем данных для проверки


### 4.2 Оптимизация ресурсов через гибридный подход

Для обеспечения работы системы на доступном оборудовании применена архитектура гибридного использования моделей разного размера.

**Таблица 7. Гибридная архитектура моделей**

| Задача | Традиционный подход | Гибридный подход | Экономия |
|--------|---------------------|------------------|----------|
| **Генерация контента** | Llama 3.1 8B (4.7GB) | Llama 3.1 8B (4.7GB) | - |
| **Валидация утверждений** | Llama 3.1 8B (4.7GB) | SentenceTransformer (118MB) | 97.5% памяти |
| **Извлечение утверждений** | Llama 3.1 8B (4.7GB) | Llama 3.1 8B (4.7GB) | - |
| **Анализ оглавления** | Llama 3.1 8B (4.7GB) | Llama 3.1 8B (4.7GB) | - |

**Использование памяти:**

| Компонент | Размер | Когда загружен |
|-----------|--------|----------------|
| Llama 3.1 8B | 4.7GB RAM | Только во время генерации |
| SentenceTransformer | 118MB RAM | Постоянно (для валидации) |
| Контекст генерации | ~20MB | Во время генерации |
| Кэш страниц | 50-100MB | Во время обработки |
| **Пиковое использование** | **~5GB** | Во время генерации |
| **В режиме ожидания** | **~200MB** | Между запросами |

**Сравнение с традиционным подходом:**

| Метрика | Традиционный | Гибридный | Улучшение |
|---------|--------------|-----------|-----------|
| Пиковая память | 8-12GB | 5GB | 58% сокращение |
| Память в ожидании | 4.7GB | 200MB | 96% сокращение |
| Скорость валидации | ~30 секунд | ~8 секунд | 3.75x ускорение |
| Минимальные требования | 16GB RAM | 8GB RAM | 2x доступнее |

**Архитектурные преимущества:**

1. **Масштабируемость**: Возможность обработки 5+ параллельных запросов на рекомендуемом оборудовании
2. **Доступность**: Работа на стандартных рабочих станциях без специализированного оборудования
3. **Эффективность**: Оптимальное использование ресурсов для каждой задачи

## Архитектурные принципы и паттерны

**Применённые архитектурные паттерны:**

1. **Pipeline Pattern** - Многофазная обработка с четкими границами между этапами
2. **Batch Processing** - Группировка данных для эффективной обработки
3. **Parallel Processing** - Одновременная обработка независимых задач
4. **Caching** - Сохранение результатов дорогих операций (offset, TOC)
5. **Strategy Pattern** - Множественные стратегии дедупликации и поиска offset
6. **Hybrid Architecture** - Комбинация разных моделей для разных задач

**Ключевые архитектурные решения:**

1. **TOC-based вместо Chunk-based** - Семантическое понимание структуры книги
2. **Автоматический offset** - Универсальность без ручной настройки
3. **Многофазная генерация** - Качество через структурированный подход
4. **Дедупликация концепций** - Устранение избыточности
5. **Гибридные модели** - Оптимизация ресурсов
6. **Валидация против источника** - Предотвращение галлюцинаций

**Результат:** Производственная система, готовая к внедрению в образовательный процесс КФУ.
